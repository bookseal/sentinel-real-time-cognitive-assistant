## Claude Code 와 같이 고민한 설계 결정들

### 1. Gradio vs Streamlit — "왜 Gradio?"

**고민 시점:** 이슈 #11 만들기 직전

**논의 내용:**
- Streamlit 은 "매 인터랙션마다 스크립트 재실행" 모델 → 실시간 스트리밍에 구조적으로 부적합
- Gradio 는 `streaming=True` 가 네이티브 → 500ms 마다 콜백, 코드 한 줄
- Streamlit 에서 마이크 쓰려면 `streamlit-webrtc` (서드파티, WebRTC 프로토콜 직접 다뤄야 함)
- 결론: 실시간 오디오 = Gradio 가 압도적

**면접 답변:** "실시간 오디오 스트리밍이 핵심이라 Gradio 선택. Streamlit 은 re-run 모델이라 streaming 에 구조적 한계."

### 2. 전역 변수 vs gr.State — "왜 State?"

**고민 시점:** 이슈 #20 (지속 알림)

**논의 내용:**
- 전역 변수 (`alert_state = {}`) 쓰면 간단하지만 탭 간 크로스토크 발생
- C 의 thread-local storage 와 같은 개념이 `gr.State`
- 각 브라우저 탭 = 독립 세션, 독립 상태
- `gr.State` 는 inputs/outputs 로 전달 — 콜백이 받고 return 으로 돌려줌

**면접 답변:** "전역 변수는 다중 사용자에서 상태가 섞입니다. gr.State 는 세션별 격리를 제공해서 C 의 pthread_key_create 와 같은 역할."

### 3. 감도 슬라이더 vs VAD — "왜 순서를 바꿨나?"

**고민 시점:** VAD 구현 시작 직전

**논의 내용:**
- VAD (Silero) 작업 시작했다가 멈춤
- "60/70 dB 임계값이 환경마다 다르다" 는 문제가 더 급함
- 카페에서는 배경 소음이 이미 55dB → 살짝만 말해도 알림
- 슬라이더 하나로 이 문제 즉시 해결, VAD 없이도 실용적
- 시니어 원칙: "새 기능 추가보다 기존 기능이 제대로 동작하게 만드는 게 먼저"

**면접 답변:** "기술적으로 멋진 것(VAD)을 먼저 하려다가, 사용자 관점에서 더 급한 것(감도 조절)을 먼저 했습니다. 결과적으로 3줄 코드 변경으로 모든 환경에서 쓸 수 있게 됐습니다."

### 4. 파일 분리 — "언제, 왜 나누나?"

**고민 시점:** VAD 추가 직전 (이슈 #22 준비)

**논의 내용:**
- app.py 가 100줄 넘어가면서 norminette 한계 접근
- 기준: "줄 수" 가 아니라 "같이 바뀌는 것끼리"
- `compute_volume_db` + `resample_to_16k` = 신호 처리 (오디오 포맷 바뀌면 같이 바뀜)
- `generate_gauge_html` + `process_audio` = 앱 로직 (UI 바뀌면 같이 바뀜)
- import 방향: app.py → audio_logic.py (단방향만)

**면접 답변:** "같은 이유로 변경되는 함수끼리 같은 파일에 두었습니다. 신호 처리와 UI 오케스트레이션의 변경 이유가 다르니까 분리. import 는 한 방향만."

### 5. `original_dtype` 버그 — "리뷰로 잡은 실제 버그"

**고민 시점:** PR #17 리뷰

**논의 내용:**
- `numpy.mean()` 이 int16 → float64 로 dtype 을 바꿔버림
- 그 뒤 `np.issubdtype(audio.dtype, np.integer)` 가 False → 정규화 스킵
- 결과: dB 값이 비정상적으로 높게 계산 (103dB 등)
- 수정: `original_dtype = audio.dtype` 를 mean() 전에 저장
- Claude 의 커스텀 review 스킬이 4축 리뷰 중 발견

**면접 답변:** "numpy.mean() 이 dtype 을 암묵적으로 변환하는 걸 몰랐습니다. 코드 리뷰에서 발견해서 mean 전에 원본 dtype 을 보존하는 한 줄로 수정."

### 6. 시니어의 라이브러리 도입 루틴

**고민 시점:** Silero VAD 탐색

**논의 내용:**
- 시니어도 새 라이브러리는 모름. 차이는 "알아내는 루틴"
- 1단계: 문제 정의 ("사람 목소리 vs 잡음 구분")
- 2단계: 용어 확인 ("VAD" 라는 이름)
- 3단계: 직접 구현 가능? (에너지 기반 = 우리 이미 함, 한계 있음)
- 4단계: Build vs Buy (정확도 필요 → ML 모델)
- 5단계: 후보 비교 (Silero 1.8MB ONNX vs pyannote 수백MB)
- 6단계: REPL 에서 최소 동작 확인 (`model(tensor, 16000)`)
- 7단계: 함수로 감싸서 통합

**면접 답변:** "문제 정의 → 용어 → 직접 구현 검토 → 라이브러리 비교 → REPL 테스트 → 통합. 이 순서를 밟았습니다."

### 고민 2: "with 문이 뭐지?" — C 개발자의 Python 첫 만남

**상황:** Gradio 의 `with gr.Blocks() as app:` 을 처음 봤을 때 완전히 생소했음.

**고민 과정:**
1. "이게 뭐하는 문법이야?" — C 에는 없는 패턴
2. Claude 가 C 비유로 설명: `goto cleanup;` 패턴의 자동화 = RAII
3. "근데 Gradio 는 리소스 정리가 아니라 컴포넌트 등록 용도잖아?" — 내가 반문
4. Claude: "맞다. Gradio 는 `with` 를 창의적으로 응용한 것. `__enter__` 에서 전역 '현재 부모' 를 설정하고, 블록 안의 컴포넌트가 자동 등록"
5. C 의사코드로 번역해서 확인:
   ```c
   Blocks *current_parent = NULL;
   blocks_enter(app);          // current_parent = app
       markdown_new("hello");  // current_parent 에 자동 등록
   blocks_exit(app);           // current_parent = NULL
   ```

**발표 포인트:** "C 개발자로서 Python 의 context manager 가 낯설었지만, C 의 `goto cleanup` 패턴과 대응시켜 이해했습니다. 특히 Gradio 가 리소스 정리가 아닌 컴포넌트 등록 용도로 `with` 를 쓰는 점이 흥미로웠고, 이건 API 설계의 창의적 응용이라는 것을 배웠습니다."

### 고민 3: "callback 이 뭐지?" — 이미 알고 있었던 것

**상황:** `audio_input.stream(fn=process_audio, ...)` 에서 callback 개념이 낯설게 느껴짐.

**깨달음:** C 에서 2년 넘게 쓰던 것과 같았음
- `qsort(arr, n, sizeof(int), compare)` — compare 가 callback
- `signal(SIGINT, handler)` — handler 가 callback
- `pthread_create(&tid, NULL, worker, NULL)` — worker 가 callback

**차이점:** C 는 함수 포인터로 명시, Python 은 함수가 일급 객체라 그냥 이름만 넘김 (괄호 없이!)

**발표 포인트:** "처음엔 callback 이라는 용어에 막혔지만, C 의 함수 포인터와 정확히 같은 개념이었습니다. `fn=process_audio` 에서 괄호를 안 붙이는 게 핵심 — C 에서 `register_callback(my_func)` 와 같습니다. 언어가 다르면 이름이 달라서 같은 개념도 낯설게 느껴진다는 걸 배웠습니다."

### 고민 4: "inputs=[audio_input] 이 왜 자기 자신?" — API 설계 비평

**상황:** `audio_input.stream(inputs=[audio_input])` 에서 자기 참조가 이상하게 보임.

**내 질문:** "이벤트 소스와 데이터 소스가 같은 건 이상하지 않나?"

**Claude 와 논의:**
- 버튼 클릭 예시: `submit_btn.click(inputs=[name_input, email_input])` — 이벤트는 버튼, 데이터는 텍스트박스 3개. 이 경우엔 명확히 다름
- Gradio 가 **일관성** 을 선택: 모든 경우에 inputs 를 명시적으로 요구. 단순 케이스에서 장황해 보이지만 복잡한 케이스에서는 명확
- "API 가 이상해 보이면 먼저 '왜 이렇게 설계됐을까' 물어봐라" — 나쁜 설계일 수도, 내가 모르는 케이스를 커버하는 일반성일 수도

**발표 포인트:** "API 가 이상하게 느껴질 때 '이건 왜 이래?' 를 물어보는 습관이 있습니다. 이 경우 이벤트 소스와 데이터 소스를 분리한 설계 결정이었고, 버튼+텍스트박스 같은 다중 입력 케이스를 보고 이해했습니다. API 디자인의 trade-off (일관성 vs 편의성) 를 생각하는 계기였습니다."

### 고민 5: "48000, 24000 이 뭔지 모르겠다" — 도메인 지식 공백 인정

**상황:** print-debugging 으로 `(48000, array(shape=(24000, 2), dtype=int16))` 을 봤는데 숫자의 의미를 전혀 모름.

**솔직한 고백:** "나는 이것들 보고 뭔지 전혀 몰랐다. 여기서 멈추게 된다."

**Claude 의 프레임:**
- 48000 = sample rate (초당 48,000 번 측정)
- 24000 = 48000 × 0.5초 = 한 청크의 샘플 수
- int16 = 각 샘플의 진폭, -32768~32767

**배운 메타 교훈:**
- "막힌 건 실력 부족이 아니라 도메인 지식의 공백"
- 프로그래밍 지식 ≠ 도메인 지식. 둘은 독립적
- 시니어도 새 도메인에서 멈춤. 차이는 "필요한 만큼만 빠르게 배운다"

**발표 포인트:** "오디오 도메인이 처음이라 sample rate, chunk size 같은 기본 개념도 몰랐습니다. 하지만 print-debugging 으로 실제 데이터를 관찰하고, '24000 ÷ 48000 = 0.5초' 처럼 역산하며 이해했습니다. 모르는 도메인에서 멈추는 건 자연스러운 것이고, 필요한 만큼만 빠르게 배우는 게 시니어 방식이라는 걸 배웠습니다."

### 고민 7: "이 f-string 너무 복잡한데 진짜 표준?" — 코드 미감

**상황:** `return f"sr={sample_rate}, shape={audio.shape}, dtype={audio.dtype}"` 이 빽빽하게 느껴짐.

**내 질문:** "진짜 이게 표준이야?"

**Claude 의 답:** "맞지만 더 나은 게 있다" → `f"{x=}"` 디버그 문법 소개

**발표 포인트:** "코드가 장황해 보여서 '이게 정말 표준인가' 질문했더니 더 깔끔한 Python 3.8+ 문법을 알게 됐습니다. 기존 코드에 불편함을 느끼는 감각이 코드 품질 개선의 출발점이라고 생각합니다."

### 고민 10: "git checkout 이 세 가지 일을 한다고?" — 도구 이해

**상황:** `git checkout -- requirements.txt` 명령이 왜 파일을 되돌리는지 이해 안 됨.

**깨달음:**
- `git checkout` 은 3가지 다른 일을 하는 **오버로딩된 명령** (git 의 인정된 설계 실수)
- git 2.23+ 에서 분리: `git switch` (브랜치) + `git restore` (파일)
- "checkout" = 도서관에서 책 빌리기. 원어민에게도 파일 되돌리기는 어색

**발표 포인트:** "git 의 표면적 명령어 뒤에 있는 설계 의도를 이해하려 했습니다. `checkout` 이 3가지 역할을 하는 건 공식적인 설계 실수이고, 그래서 `switch` 와 `restore` 로 분리됐다는 역사를 알게 됐습니다. 도구를 쓸 때 '왜 이렇게 생겼는지' 를 궁금해하는 습관이 있습니다."

### 고민 11: "Silero VAD README 예제가 우리 상황에 안 맞는데?" — 라이브러리 도입

**상황:** README 의 `get_speech_timestamps(wav, model)` 은 파일 기반. 우리는 실시간 청크.

**시니어 판단 과정:**
1. README 의 high-level API (`get_speech_timestamps`) 는 파일용 → 안 맞음
2. 하지만 그 안에서 쓰는 low-level API (`model(tensor, sr)`) 는 쓸 수 있음

4. "README 를 그대로 안 쓰고 모델만 빌려 쓰기" — 라이브러리의 추상화 수준을 선택하는 것

**발표 포인트:** "라이브러리 README 의 예제가 우리 유즈케이스에 안 맞았습니다. high-level API 대신 model 을 직접 호출하는 low-level 접근을 선택했습니다. REPL 에서 먼저 테스트하고, 동작을 확인한 뒤 통합하는 과정을 거쳤습니다."
