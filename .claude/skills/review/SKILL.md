# Code Review Skill

현재 작업 중인 변경사항(git diff)을 체계적으로 리뷰한다. 버그 · 보안 · 성능 · 컨벤션 4개 축으로 검사하고, 각 발견 항목에 심각도를 부여한다.

## 1. 변경사항 수집

다음 명령을 **병렬로** 실행해 리뷰 대상을 파악한다.

1. `git status` — 수정/추가/삭제된 파일 목록
2. `git diff` — unstaged 변경 내용
3. `git diff --staged` — staged 변경 내용
4. `git diff main...HEAD` — 브랜치가 main에서 분기된 이후 모든 변경

변경사항이 없으면 즉시 종료하고 "리뷰할 변경사항이 없습니다"라고 보고한다.

큰 파일이 변경된 경우 `Read` 툴로 해당 파일 전체를 읽어 컨텍스트를 확보한다. diff만으로는 주변 코드 흐름을 놓치기 쉽다.

## 2. 검사 축

### 🐛 Bugs (버그/정확성)
- Off-by-one, null/undefined 접근, 타입 불일치
- 에러 핸들링 누락 또는 과잉
- race condition, 비동기 처리 오류
- 로직 분기에서 빠진 케이스
- 리소스 누수 (파일 핸들, 커넥션, 타이머)

### 🔒 Security (보안)
- 명령어 주입, SQL 주입, XSS
- 하드코딩된 시크릿/API 키/토큰
- 검증 없는 사용자 입력이 시스템 경계를 통과
- 민감정보 로깅
- 안전하지 않은 역직렬화, 경로 조작
- OWASP Top 10 전반

### ⚡ Performance (성능)
- N+1 쿼리, 불필요한 반복문 중첩
- 큰 객체 복사, 불필요한 재계산
- 동기 I/O가 이벤트 루프를 막는 경우
- 메모리 누수 가능성 (전역 상태 누적, 리스너 미해제)
- 핫패스에서의 비효율적 자료구조 선택

### 📐 Conventions (컨벤션) — [CLAUDE.md](../../../CLAUDE.md) 기반
- **파일당 함수 7개 초과 금지**
- **함수당 30줄 초과 금지** (docstring/공백 제외)
- **함수 1개당 1책임**
- **중첩보다 평탄화** — early return 선호
- **파일이 150줄을 넘으면 분리 고려**
- Conventional Commits / 브랜치 네이밍 (`type/short-description`)
- 언어별 표준 (PEP8 / ESLint 등 프로젝트에 이미 설정된 규칙)

## 3. 심각도 레벨

| 심각도 | 기호 | 기준 | 처리 |
|---|---|---|---|
| **Critical** | 🔴 | 프로덕션 장애, 데이터 손실, 보안 침해로 직결 | **머지 차단** |
| **High** | 🟠 | 명백한 버그, 악용 가능한 취약점, 주요 컨벤션 위반 | 머지 전 수정 필수 |
| **Medium** | 🟡 | 엣지 케이스 버그, 성능 저하, 리팩토링 필요 | 이번 PR 또는 후속에서 수정 |
| **Low** | 🟢 | 스타일, 네이밍, 사소한 개선 | 선택적 |
| **Info** | 🔵 | 칭찬할 점, 관찰 메모 | 참고 |

## 4. 리뷰 출력 형식

다음 구조로 결과를 출력한다. 발견 항목이 없는 섹션은 "없음"으로 표기한다.

```
## 📋 Review Summary

- **대상**: <브랜치명> (<파일 N개, +X -Y 라인>)
- **Critical**: N | **High**: N | **Medium**: N | **Low**: N
- **전체 판정**: ✅ 머지 가능 / ⚠️ 수정 후 머지 / ❌ 머지 차단

---

## 🐛 Bugs
### 🔴 [Critical] <한 줄 제목>
- **위치**: [path/to/file.py:42](path/to/file.py#L42)
- **문제**: <구체적인 설명>
- **제안**:
  ```language
  <수정 예시>
  ```

## 🔒 Security
...

## ⚡ Performance
...

## 📐 Conventions
...

---

## 💡 추가 관찰
- <전반적인 인상, 구조적 제안, 칭찬할 점 등>
```

## 5. 리뷰 원칙

- **구체적으로 지적한다.** "이 코드는 좀 이상함"이 아니라 "34줄에서 `user.email`이 None일 수 있는데 직접 접근하고 있음"처럼 파일:라인을 반드시 명시한다.
- **수정 예시를 보여준다.** 말로만 설명하지 말고 교체 가능한 코드 조각을 붙인다.
- **근거를 댄다.** "CLAUDE.md의 함수 30줄 규칙 위반", "OWASP A03 주입 취약점" 등 판단 기준을 인용한다.
- **칭찬도 한다.** 잘 짜인 부분은 🔵 Info로 언급한다. 리뷰는 질책이 아니라 대화다.
- **불확실하면 질문한다.** 의도를 모르면 추측해서 비판하지 말고 "이 분기의 의도는 X인가요?"라고 적는다.
- **스코프를 지킨다.** diff에 없는 코드까지 끌어와서 리팩토링을 요구하지 않는다. 변경된 라인과 그 직접 영향 범위에 집중한다.

## 6. 완료 조건

리뷰가 끝나면:
1. Summary 블록에서 전체 판정(✅/⚠️/❌)을 명확히 제시
2. Critical·High 항목이 있으면 사용자에게 "수정하시겠습니까?"라고 묻는다
3. 사용자가 수정을 요청하면 해당 항목부터 순서대로 고친다 (이때는 리뷰 스킬 종료, 일반 편집 모드로 전환)
